# AGENTS.md

## 职责

本文件是 AI 进入仓库后的第一入口，包含强约束、任务分级、工程风格、项目专属红线和证据收口。

## 必读

进入仓库后，必须先读取：

1. `README.md`：项目总说明、文件地图、内容生产流程和验证口径。
2. 本文件：AI 协作约束。
3. `xiaohongshu-cute-knowledge-prd.md`：项目 V1 需求文档，是账号定位、内容范围和首篇样稿要求的事实来源。

## AI 协作材料

- 本仓库不保存临时 AI 协作过程材料或历史过程归档。
- 复杂协作过程材料必须由用户明确指定仓库外目录或外部工程承接；本仓库只保留业务文档、项目说明、内容方案和必要验证脚本。
- 一次性草稿、临时 Prompt 试验、未采用标题池和过程记录默认不提交到仓库，除非用户明确要求保留。

## 任务分级

### 只读核查

只允许读取文件、搜索文本、查看状态、查看差异、读取日志、读取配置和调用不会改变状态的查询接口。

只读核查不得写文件、写数据库、执行迁移、安装依赖、重启服务、提交 Git 或推送 Git。

当前仓库暂无数据库。若后续接入热点库、评论库或数据看板，数据库只读查询前必须说明环境、库名和影响范围；只读澄清优先只执行普通 `SELECT`。如数据库支持只读事务，可用 `START TRANSACTION READ ONLY`（或等价语法）后执行 `SELECT`，最后用 `ROLLBACK` 结束只读事务。这里的 `ROLLBACK` 只表示结束只读查询事务，不是业务数据回滚；事务内禁止任何写入、迁移、DDL、DML 或状态变更语句。

### 小改动

允许修改当前任务直接相关的 Markdown 文档、内容样稿、Prompt 草案、检查清单或验证脚本，并运行本地匹配范围验证。

小改动不包含数据库写入、迁移、远端服务状态变化、依赖安装、真实业务写接口调用、账号发布或生产发布。

### 真实联调

涉及真实接口调用、数据库写入、浏览器自动化、图像生成 API、小红书平台写入、热点采集接口、自动化发布工具或跨系统端到端验证时，属于真实联调。

真实联调执行前必须说明环境、对象、影响范围和验证证据。

本项目当前默认不做真实联调。创作者使用 GPT-image2 生成图片属于仓库外人工动作，AI 只产出画面描述和 Prompt 草案，不能默认代替用户调用图像生成或发布平台。

### 生产发布

当前仓库没有生产发布流程。

任何小红书账号发布、平台写入、自动评论、自动私信、自动采集用户数据、自动定时发布或远端自动化任务，都必须获得用户明确授权。执行前必须说明账号、平台、内容对象、影响范围、可回查证据和失败后的处理方式。

AI 不允许使用 Git 回滚脚本或回滚命令修复流程错误（包括 `git reset`、`git reset --hard`、`git checkout <commit> -- <path>`、`git restore`、`git revert` 等）。发生错误时必须停止，输出当前分支、HEAD、工作区状态、已执行命令和可选人工恢复方案。

## 必须

- 写文件前，先说明要改哪些文件和目标。
- 改动范围必须收敛，只处理当前任务。
- 修改前必须界定本次目标行为和废弃行为，只沿当前任务直接相关的文档、样稿、Prompt、检查清单和验证脚本修改。
- 凡是准备执行删除、清空、覆盖、替换、移动、撤回、回滚等可能破坏现状的操作前，必须先只读查询确认目标、范围和影响；涉及历史数据、日志、配置、远端目录等状态性内容时，必须停止并说明，不能自由发挥直接执行。
- 如果旧逻辑已被废弃，必须删除旧实现、旧分支、旧测试期兜底、旧文档描述和相关测试，不保留新旧双轨；只有用户明确要求兼容期时，才允许保留旧逻辑，并必须说明兼容范围、触发条件和后续删除点。
- 说明项目事实、内容范围、问题、修改建议时，尽量给出文件和行号。
- 改动后必须说明改了什么、验证了什么、还剩什么风险。
- 涉及数据库、迁移、远端服务、真实接口调用、平台发布或图像生成 API 时，先说明环境和影响范围。
- 结论必须区分事实、推断和未验证风险。
- 生成内容方案时，必须对照 PRD 的渠道、主题、画幅、页数、语气和视觉风格约束。
- 用户给出选题并要求走外部 LLM 流程时，必须先基于 `templates/external-llm-copy-prompt-template.md` 生成专用提示词，再等待用户贴回粗稿后定稿。
- 每次定稿前必须读取 `templates/humanize-copy-checklist.md`，检查并降低 AI 腔、八股感和说明书感。
- 生成算法科普图文结构前，必须优先使用 `templates/algorithm-xhs-infographic-generator-template.md`，固定解释骨架但不固定具体生活例子。
- 每次生成或改写 Image2 Prompt 前必须读取 `templates/image2-infographic-prompt-guide.md`，避免信息过载、文字翻车和风格混杂。

## 工程风格

### 语言与文本（违反 = 返工）

1. 代码标识符（模块、类、函数、变量、表名、字段名）必须使用英文。
2. 代码注释、docstring、面向业务的说明文字必须使用中文。
3. 中文文档必须使用 UTF-8 编码保存。
4. Git commit message 必须使用中文，前缀必须是以下之一，禁止其它前缀：`feat:`、`fix:`、`docs:`、`refactor:`、`test:`、`chore:`。

### 业务注释（违反 = 返工）

5. 业务注释统一使用中文 `#` 注释（或当前语言的单行注释语法）。
6. 文件职责注释放在文件顶部可见区域，说明当前文件承担的业务边界。
7. 方法职责或逻辑概览注释放在函数或方法定义上方。
8. 函数内部大步骤注释放在对应代码块上方，说明业务阶段、状态流转或关键处理顺序。
9. 关键取值、关键分支、关键校验、异常语义、硬规则和软约束来源，必须在就近位置用注释说明业务原因。
10. 同行注释只用于必须贴近单行表达式说明的关键取值或关键分支；默认优先写在代码上方。
11. docstring 不承载业务规则、状态流转、硬规则、软约束或长段逻辑说明。
12. 注释解释业务逻辑、字段含义、状态流转、异常语义和规则来源，不解释显而易见的语法。
13. 注释只写当前有效事实，不写历史过程、不写新旧对照、不描述已废弃逻辑。
14. 在既有逻辑中的点状修改，必须在修改点附近说明该改动改变的业务行为、数据流或状态边界。
15. 复杂链路注释必须服务人工顺调用链查证，优先放在正式入口、批量触发点、并发边界、外部接口调用点、状态回写点、结果拆分或合并点、业务主键映射点。
16. 注释不得作为兼容说明保留已废弃入口、旧分支、兜底路径或双轨行为；废弃行为必须在代码和测试中同步清理。

### 代码风格（违反 = 返工）

17. 导入语句必须使用从项目模块根开始的绝对路径。禁止任何形式的相对导入。
18. 禁止创建以下文件名的模块：`utils`、`common`、`helpers`、`misc`、`tools`。
19. 新增源码文件名必须包含动词或具体名词。禁止仅以形容词或泛指词命名。

### 测试命名（违反 = 返工）

20. 测试函数名禁止包含：数字日期、任务编号、审查日期。
21. 测试函数名必须描述被测行为或被测场景，不描述实现细节。

## 内容风格

- 面向普通小红书用户，不默认读者有算法或 AI 背景。
- 先讲生活问题，再讲背后机制，不从术语定义开头。
- 语气像朋友解释给朋友听，轻松但不糊弄，简单但不幼稚。
- 每篇只解释一个核心算法或机制。
- 技术术语必须少量、后置，并配生活化解释。
- 中性解释平台和算法机制，不用情绪化判断替代机制分析。
- 不为了可爱牺牲准确性；复杂处必须说明“这是简化版理解”。
- 不写 AI 八股文：少用排比、总分总、上价值结尾和“首先/其次/最后”。
- 抽象词必须落到具体场景、动作或感受上，宁可写“脑子一下打结”，不要写“降低认知成本”。
- 允许适量碎碎念、小停顿和不完美句子，但不能变成夸张人设或影响准确性。
- 知识解释可以温柔收住，不强行升华。

## 架构红线

### 项目定位

- 本项目是“小红书可爱图解知识账号”的内容策划和样稿仓库。
- 本项目负责用软萌治愈、贴纸手账风解释 AI / 算法小机制。
- 本项目不负责硬核技术教学、多平台运营、自动发布、真实用户数据采集或高成本 IP 世界观建设。

### 内容边界

- 早期选题必须围绕 AI / 算法 / 推荐 / 排序 / 匹配 / 优化 / 平台机制，不得直接扩成全领域百科。
- 首批内容比例默认遵循 PRD：单算法 / 单概念解释 60%，AI 小助手背后的机制 30%，平台小机关 10%。
- 平台小机关属于后续进阶系列，必须在基础概念内容跑通后再加大比例。
- 财经、历史系统故障等方向只作为后续扩展，不进入 MVP，除非用户明确改范围。

### 入口与操作限制

- 正式业务入口是 Markdown 文档、内容样稿、分镜方案、Prompt 草案和复盘表格。
- 不新增自动发布入口、账号写入入口、用户数据采集入口或图像生成 API 调用入口，除非用户明确授权并更新 README 的环境口径。
- 使用小红书相关 Skill 时，只能辅助选题、标题、正文、分镜、画面描述、Prompt 草案和复盘，不能让 Skill 改写账号定位或自动扩大主题范围。
- 外部 LLM 只负责生成粗稿，最终定稿必须回到本项目 PRD、README 和 AGENTS 约束下审查。
- 算法科普内容不能硬编码某个生活例子。排课、排班、座位、路线、小店员等都只是单篇解释载体；跨选题复用时只能复用解释骨架，不能复用具体场景。

### 文件命名约束

- 项目说明文档使用清晰英文或拼音短横线命名，例如 `xiaohongshu-cute-knowledge-prd.md`。
- 内容样稿文件名应包含主题和产物类型，例如 `cp-sat-seat-arrangement-draft.md`。
- 提示词模板放在 `templates/` 目录，文件名应包含具体用途，例如 `external-llm-copy-prompt-template.md`。
- 禁止新增 `temp.md`、`draft.md`、`notes.md`、`misc.md` 这类泛指文件名，除非用户明确要求且说明用途。
- 临时草稿默认放在用户指定的仓库外目录；未指定时不落盘。

### 内容完整性

- 外部 LLM 提示词必须要求输出标题备选、核心解释、生活化比喻、按算法步骤和难易程度动态确定的分镜、小红书正文、标签和准确性风险提醒；只有用户明确要求节奏对比时，才额外输出 2 页版或 4 页版。
- 每篇样稿必须包含标题备选、图文分镜、每页配文、每页画面描述、每页 GPT-image2 Prompt 草案、小红书正文和标签。
- 每篇样稿必须包含去 AI 味检查记录，说明是否处理了节奏、抽象词、总结腔和过度工整问题。
- 每篇算法科普样稿必须明确算法名、生活例子、每页步骤、阅读路径和最后总结图；不能只产出可爱插画场景。
- 算法科普页数必须按算法步骤和难易程度动态决定；不得机械套用 5 页，也不得退回旧的 2 页/4 页结构。无论页数多少，都必须保留总结页或大体了解图。
- 多图内容必须保持角色、场景、道具、色彩和叙事连续性。
- 每页只能承载一个小步骤或一个关键变化。
- 最后一页必须收束成一句可记住的总结。
- 每篇必须包含互动问题。
- Prompt 必须明确画幅为 3:4，并体现软萌治愈 + 贴纸手账。
- Image2 Prompt 必须拆成主标题、短副标题、页面结构、配色、单一主风格、生成要求和禁止项；每页默认不超过 5 个视觉模块。
- Image2 Prompt 里的中文文字必须少而短；长句说明默认后期手动加，不要求模型直接生成。

### 准确性与风险

- 不把简化比喻写成严格技术定义。
- 不使用绝对化、夸大化、制造恐慌或强立场表达解释平台机制。
- 涉及 AI 能力、平台推荐、动态定价、风控、用户画像等敏感机制时，必须使用“通常”“可以理解为”“简化来说”等边界表达。
- 如内容基于热点、新闻、平台规则或模型能力，必须先核查最新信息来源；无法核查时明确标注为未验证风险。

### 测试约束

- 文档改动后必须检查模板占位符是否残留。
- 内容样稿改动后必须核对 PRD 约束：小红书、AI / 算法、纯图文、3:4、软萌治愈 + 贴纸手账、中性解释、生活化表达。
- 内容样稿定稿后必须核对去 AI 味约束：无“首先/其次/最后”式八股结构、无空泛大词堆叠、无强行升华结尾。
- Image2 Prompt 定稿后必须核对：主题 3 秒内可见、结构 10 秒内可懂、文字少、留白足、模块少、主风格单一。
- 如新增脚本，必须提供只读或本地文件级验证命令；脚本不得默认访问远端、写平台或修改用户数据。

## 证据口径与收口

### 小改动证据

- 本次修改文件。
- 本次关键 diff。
- 匹配范围验证结果。
- `git status` 状态；如果当前目录不是 Git 仓库，必须说明。

### 内容样稿证据

- 选题和对应核心机制。
- 页数、画幅和视觉风格。
- 每页承接关系。
- 标题、正文、标签和互动问题。
- 准确性边界和未验证风险。

### 真实联调证据

真实联调默认不执行。若用户明确授权，证据至少包含：

- 调用平台或接口。
- 请求关键字段。
- 响应关键字段。
- 生成或发布对象。
- 时间、账号或任务标识。
- 可回查截图、日志或平台记录。

### 临时产物

- 一次性排查脚本、临时提示词试验、浏览器自动化截图和临时测试记录默认不提交。
- 复杂协作过程材料默认不提交；如需保存，必须由用户明确指定仓库外目录或外部工程。
- 可复用验证脚本可保留在项目的临时验证目录，但必须有明确用途和稳定入口。
- 临时产物是否提交，以用户明确要求为准。

### 收口输出

完成任务后必须说明：

- 改了什么。
- 验证了什么。
- 没有验证什么。
- 剩余风险是什么。
- 是否已提交和推送。

- 读取中文文件时注意指定正确的编码，避免乱码。
